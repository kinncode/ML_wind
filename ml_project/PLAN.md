# BSMI 測風塔 — 機器學習預測專案規劃

> 用途：練習與作品集｜算力假設：一般筆電 CPU、16 GB RAM
> 本文件中的所有統計數字，都是從實際資料抽樣計算出來的（17 個月、71,261 筆 10 分鐘紀錄），不是估計值。

---

## 1. 先講清楚：這不是三個專案

三個資料夾是**同一座測風塔的三個時段**，不是三份不同的資料。合併後是一份連續的長期觀測：

| 項目 | 內容 |
|---|---|
| 站點 | BSMI 100 m 測風塔（Campbell CR1000，TOA5 格式） |
| 期間 | 2016-03-16 ~ 2021-10-31，共 66 個月 |
| 解析度 | 1 Hz（逐秒），約 1.6 億筆、14 GB |
| 缺漏 | 整月缺 2020-03、2021-06；月內另有零星中斷（2016-07/08、2020-11 較明顯） |
| 感測器 | 風速 100E / 100W / 69W / 38W（m/s）<br>風向 97 / 35（deg）<br>氣溫 95 m、相對濕度 95 m、氣壓 93 m |
| 資料品質 | 抽樣檢查 **98.8% 的 10 分鐘格通過品管**，無 NAN |

**這其實是好消息。** 66 個月連續資料、四個高度、雙備援 100 m 風速計、逐秒解析度 —— 這種等級的測風塔資料通常拿不到。

**但要先接受一個限制：資料裡沒有發電量。** 這是純氣象塔，沒有 SCADA、沒有功率輸出。所以「功率曲線建模」「風機故障預測」這類題目做不了，必須從氣象量本身找預測題目。

## 2. 決定題目可行性的四個資料特性

### (1) 季節訊號極強，強到會主導一切

| 月份 | 100 m 平均風速 | 亂流強度 TI（中位數） | 風切指數 α | 風功率密度 |
|---|---|---|---|---|
| 1 月 | **11.46 m/s** | 0.053 | 0.060 | 1580 W/m² |
| 4 月 | 6.84 m/s | 0.060 | 0.052 | 509 W/m² |
| 7 月 | **5.84 m/s** | 0.098 | 0.133 | 222 W/m² |
| 10 月 | 9.10 m/s | 0.061 | 0.063 | 1311 W/m² |

冬夏風功率密度差 **7 倍**。這代表兩件事：

- 訓練集**必須包含完整年份**，否則模型學到的是季節而不是動力。
- 任何沒有做季節分層評估的結果都不可信 —— 光靠「現在是幾月」就能解釋掉大半變異。

冬季 TI 只有 0.05、風切 α 只有 0.06 —— 這是接近中性、充分混合的強季風條件，風況品質相當於離岸場址。

### (2) Persistence 基準線非常強（這決定了你的題目該做幾小時）

以「下一時刻 = 現在」當預測（persistence），實際誤差如下：

| 預測提前量 | RMSE (m/s) | MAE (m/s) | 相關係數 r |
|---|---|---|---|
| 10 分鐘 | **0.601** | 0.423 | 0.9935 |
| 1 小時 | 1.501 | 1.090 | 0.9593 |
| 3 小時 | 2.474 | 1.847 | 0.8894 |
| 6 小時 | 3.400 | 2.548 | 0.7911 |
| 12 小時 | 4.457 | 3.296 | 0.6413 |
| 24 小時 | 5.151 | 3.627 | 0.5224 |

對照組：全期風速標準差 = **5.257 m/s**（也就是「永遠猜平均值」的 RMSE）。

**這張表是整個規劃最重要的一張表**，因為它直接告訴你：

- **10 分鐘**：persistence 的 r 已經 0.9935。機器學習大概只能再擠出 5–10% 的改善。做這個題目要有心理準備，重點會是「證明你的模型真的有超過 persistence」而不是「拿到漂亮的 R²」。
- **3–6 小時**：persistence 開始明顯劣化，但還遠優於猜平均。**這是機器學習最有價值的區間**，建議把主力放在這裡。
- **24 小時**：persistence 的 RMSE（5.151）已經逼近猜平均（5.257），等於完全失效。單靠塔上的觀測要做 24 小時預報，物理上就缺資訊（需要數值天氣預報當輸入），這題不建議在沒有外部資料的情況下硬做。

> ⚠️ 網路上大量「風速預測 R²=0.99」的專案，其實只是把 persistence 重新包裝一次。你的作品集要贏在這裡：**永遠報告相對 persistence 的技術得分（skill score）**。

### (3) 日夜變化明顯

去除月平均後，日夜振幅：夏季 7 月 **2.89 m/s**、冬季 1 月 **3.49 m/s**。所以「小時」「一年中的第幾天」要用 sin/cos 編碼成週期特徵，不能當成數字直接餵。

### (4) 感測器健康度很好 —— 這反而限制了異常偵測題目

兩支 100 m 風速計的差異中位數只有 **0.048 m/s**（p99 = 0.352 m/s），71,261 格中只有 177 格（0.25%）被判定為配對異常、18 格疑似卡死。

代表：**「感測器異常偵測」這題目正樣本太少**，很難做出有說服力的驗證。建議降級為專案的一個附屬模組，而不是主題。

## 3. 共同基礎建設（已完成）

`preprocess.py` 已寫好並在真實資料上驗證過，處理了三個時段的格式差異（2016–2017 時間戳有引號、欄名 `Record` vs `RECORD`、2021-07 之後副檔名變 `.txt`）。

**執行方式**

```bash
pip install pandas numpy pyarrow
python preprocess.py --root "D:/ML_wind" --out "D:/ML_wind/ml_project/data"
```

**輸出**：14 GB → 約 287,000 列 × 48 欄的 parquet，**檔案約 60 MB**，可以整張塞進記憶體，後續所有建模都不用再碰原始檔。

實測效能：每月約 6–7 秒（本機 SSD）、尖峰記憶體 1.3 GB。全部 66 個月約 **8–10 分鐘**跑完。

**產出的特徵**（每 10 分鐘一列）

| 類別 | 欄位 |
|---|---|
| 風速 × 4 高度 | `_mean` `_std` `_min` `_max`(1 秒陣風) `_ti` `_gust_factor` |
| 風向 × 2 高度 | `_vecmean`（向量平均）`_sigma`（Yamartino 法）`_sin` `_cos` |
| 氣象 | `AT_95_mean` `RH_95_mean` `BP_93_mean` |
| 衍生 | `shear_alpha`（三高度對數擬合）`veer_97_35`、`air_density`（含濕度修正）、`power_density` |
| 品管 | `coverage` `flag_frozen` `flag_pair_mismatch` `flag_low_coverage` `is_valid` |

兩件刻意的設計，值得在作品集裡說明：

- **風向用向量平均，不用算術平均。** 359° 和 1° 的算術平均是 180°，完全錯誤。
- **TI 和陣風因子只在平均風速 > 3 m/s 時計算。** 低風速下分母趨近 0，TI 會爆炸成無意義的大數，這是 IEC 61400 的慣例。

## 4. 專案清單（依建議順序）

### 🥇 專案 A：多時程風速預測 — 主力題目

| | |
|---|---|
| **目標** | 預測 `WS_100_mean` 在 t+10min / +1h / +3h / +6h 的值 |
| **輸入** | 過去 3 小時的風速/風向/TI/風切/氣溫/氣壓 lag 特徵、氣壓變化率（3h tendency）、時間週期編碼 |
| **必備基準** | ① Persistence ② 氣候平均（該月該時的歷史均值）③ 線性迴歸 |
| **主模型** | LightGBM（每個時程各訓一個模型，direct multi-horizon） |
| **選配** | 淺層 LSTM 或 TCN（序列長度 18，隱藏維度 64；CPU 上約數分鐘/epoch，可行但非必要） |
| **指標** | RMSE、MAE、**Skill Score = 1 − RMSE_model / RMSE_persistence** |
| **合理預期** | 10 min 改善 5–10%；1 h 改善 10–15%；3–6 h 改善 20–35%（需實測驗證） |
| **工作量** | 3–5 天 |

**為什麼這題值得做**：它是風能領域的標準問題，任何面試官都聽得懂；而且因為 persistence 基準線很強，能不能誠實地打敗它，直接反映你的方法學功力。

**加分做法**：氣壓變化率（`BP_93_mean` 的 3 小時差分）是天氣系統移動的直接指標，在 3–6 小時時程上通常是最有價值的單一特徵。這是物理直覺轉成特徵工程的好例子。

---

### 🥈 專案 B：垂直外推 / 虛擬感測器 — CP 值最高

| | |
|---|---|
| **目標** | 只用 38 m（低層）的觀測，推估 100 m（輪轂高度）的風速 |
| **為什麼重要** | 這是風場開發的真實痛點：架 100 m 測風塔很貴，開發商常只有矮塔或短期光達資料，必須外推到輪轂高度。外推誤差 1% 會直接放大成發電量誤差 3%（因為功率正比於 U³）。 |
| **物理基準（已實測）** | 冪次律 α = 1/7（業界預設）：**RMSE 1.010 m/s，偏差 −0.347 m/s**<br>冪次律 α = 本站中位數：**RMSE 0.811 m/s，偏差 +0.239 m/s** |
| **主模型** | LightGBM，輸入 38 m 風速 + 風向 sin/cos + 38 m TI + 氣溫 + 濕度 + 氣壓 + 時間編碼 |
| **指標** | RMSE、MAE、偏差，以及**分風向、分季節、分穩定度的誤差分解** |
| **工作量** | 2–3 天 |

**為什麼推薦**：目標明確、基準線是有物理意義的公式（不是隨便一條線）、訓練只要幾秒鐘、故事線乾淨 ——「業界慣用的 α=1/7 在這個站有 −0.35 m/s 系統性低估，機器學習把 RMSE 從 1.01 降到 X」。這是作品集裡最容易講清楚價值的一個。

**已有的證據支撐**：本站風切隨大氣狀態變化極大（1 月中位數 α=0.060，7 月 α=0.133，夏季高 TI 時段可達 0.271）。固定 α 必然失準，這正是機器學習該切入的地方。

---

### 🥉 專案 C：極端陣風預警 — 補上分類與不平衡學習

| | |
|---|---|
| **目標** | 預測未來 1 小時內是否出現 1 秒陣風 > 20 m/s（二元分類） |
| **實際樣本分布（已實測）** | 陣風 > 15 m/s：18.8%｜**> 20 m/s：6.12%**｜> 25 m/s：1.42%｜歷史最大 37.75 m/s |
| **為什麼重要** | 25 m/s 是多數風機的切出風速；陣風預警關係到維護人員登塔安全 |
| **主模型** | LightGBM 分類（`scale_pos_weight` 處理不平衡）+ 機率校準（Platt / Isotonic） |
| **指標** | **PR-AUC**（不是 ROC-AUC —— 正樣本只有 6%，ROC 會過度樂觀）、Brier score、可靠度曲線 |
| **加值** | 畫出「預警門檻 vs 漏報/誤報」的成本權衡曲線，讓使用者自己選作業點 |
| **工作量** | 2–3 天 |

6.12% 的正樣本率是**剛剛好**的不平衡程度 —— 夠稀有到需要處理，又不至於稀有到訓不起來。

---

### 專案 D：大氣穩定度分類與風切預測（進階）

先用非監督方法（K-means 或 GMM）在 `shear_alpha` + `veer_97_35` + `TI` + `Δ(風速-氣溫)` 的空間中分出穩定度型態，再訓練分類器從基本氣象量預測型態。

已觀察到的訊號（7 月，依 TI 分層）：

| TI 區間 | 樣本數 | 風切 α 中位數 | 風向剪切中位數 |
|---|---|---|---|
| 0.00–0.05 | 1,935 | 0.050 | 3.92° |
| 0.05–0.10 | 6,653 | 0.076 | 5.57° |
| 0.10–0.15 | 3,159 | 0.217 | 5.60° |
| 0.15–0.30 | 4,950 | 0.271 | 5.18° |

**⚠️ 這裡有個混淆必須處理**：TI = σ/U，分母就是風速。實測相關係數為 corr(TI, U) = **−0.31**、corr(α, U) = **−0.27** —— 風速低時 TI 和風切都會變大。所以上表的「TI 高 → 風切大」有一部分是風速造成的共同驅動，不能直接讀成因果。

相關性只有 0.3 左右，代表這不是純粹的假象、確實還有大氣穩定度的獨立訊號，但**任何 TI 或穩定度模型都必須先跟「只用風速」的基準模型比過**，才能宣稱有加值。把這段混淆分析寫出來，本身就是作品集裡的加分內容。

工作量 3–4 天，建議在 A、B 完成後再做。

### 專案 E：長期風資源評估 MCP（需要外部資料）

風能業界的標準流程：用短期現地觀測 + 長期參考資料，推估場址的長期平均風況。

做法：只用 12 個月訓練，預測另外 54 個月，**再跟真實值比對** —— 這是難得的驗證機會，因為真實的 MCP 案子沒有答案可以對。需要額外下載 ERA5 再分析資料（免費，但要註冊 Copernicus CDS）。

因為依賴外部資料下載，建議排在最後。

## 5. 方法學紅線（時間序列最常見的錯誤）

這一節建議直接寫進你的 README，是作品集的加分項。

1. **絕對不能隨機切分。** 用時間切：訓練 2016-03 ~ 2019-12｜驗證 2020｜測試 2021。隨機切分會讓 t 和 t+10min 分屬訓練與測試集，等於直接洩漏答案。
2. **交叉驗證要用 rolling-origin（前進式）**，不能用 KFold。
3. **在切分點前後留 gap。** 特徵用到過去 3 小時，切分邊界就至少留 3 小時空白，否則驗證集的特徵包含訓練集的資料。
4. **跨資料缺口的樣本要丟掉。** 2020-03 和 2021-06 整月缺失，橫跨這些缺口的 lag 特徵和預測目標都無效。`is_valid` 欄位可以用來篩，但要在建立 lag **之後**再篩。
5. **一定要報告 skill score。** 只給 RMSE 沒有意義 —— 冬季 RMSE 2.0 是好結果，夏季 RMSE 2.0 是災難。
6. **分層評估。** 至少要分季節、分風速區間各報一次誤差。整體平均會被冬季高風速時段主導。
7. **`power_density` 不能當特徵去預測風速。** 它是從風速算出來的（0.5ρU³），直接洩漏答案。同理 `WS_100_mean` 和 `WS_100E_mean` 互為洩漏。

## 7. 環境需求

```
pandas>=1.5    numpy    pyarrow        # 資料處理
lightgbm       scikit-learn           # 主力模型
matplotlib     seaborn                # 視覺化
windrose                              # 風花圖（選配）
```

全部都是 CPU 套件。以 285,000 列 × 60 特徵的規模，LightGBM 單次訓練約 10–30 秒，超參數搜尋（100 組）約 30–60 分鐘 —— 16 GB 筆電完全足夠。

深度學習（LSTM/TCN）**不是必要的**。在這個資料規模上，梯度提升樹通常打得贏或打平，而且訓練快 100 倍、更好解釋。如果要放進作品集，建議當成「我試過並且說明為什麼不用它」的一段對照，比硬塞一個 LSTM 更有說服力。
